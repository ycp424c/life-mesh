# MCP And Tool Interface

状态：draft
最后更新：2026-06-26
职责边界：记录 LifeMesh 对 MCP 和工具接口的设计假设、能力边界和待决问题。

## 当前立场

MCP 可以作为 Agent 接入 LifeMesh 的候选接口层，但当前文档阶段不绑定实现。

## 接口设计原则

- 工具接口暴露能力，不暴露数据库。
- 每个工具声明需要的数据范围、动作风险和审计字段。
- 只读、写入、外发、执行类能力分开授权。
- 返回结果必须尽量包含来源、时间和置信度。
- 高风险工具调用必须支持用户确认。

## 候选工具族

| 工具族 | 能力 | 风险 |
|---|---|---|
| document-search | 搜索文档、返回来源片段 | 低到中 |
| memory-read | 读取任务相关记忆 | 中 |
| memory-write | 写入或更新长期记忆 | 中到高 |
| calendar-task | 查询日历、创建任务草稿 | 中 |
| contacts-context | 查询联系人和关系事实 | 高 |
| automation | 执行外部动作 | 高到极高 |

## 工具定义应包含

- 名称
- 输入参数
- 输出结构
- 需要的授权
- 访问的数据分类
- 是否写入数据
- 是否可能外发信息
- 审计事件类型
- 失败和撤销策略

## 待决问题

- 首个 MCP Server 是按数据类型拆分，还是按用户工作流拆分？
- MCP Resources、Prompts、Tools 如何映射到 LifeMesh 数据层？
- 工具调用前的用户确认由 LifeMesh 做，还是由宿主 Agent 做？
