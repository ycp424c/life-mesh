# Product Brief

状态：draft
最后更新：2026-06-26
职责边界：定义 LifeMesh 要解决的问题、目标用户、核心价值和非目标。

## 一句话

LifeMesh 是个人数据和长期记忆的上下文基础设施，让 AI Agent 能在授权、可审计、可撤销的边界内理解并协助个人生活与工作。

## 核心气质

- 个人数据主权
- 长期记忆
- AI Agent 可用的上下文基础设施
- 可追溯、可授权、可持续进入的数据底座

## 目标用户

初始目标用户是愿意长期管理个人数据，并希望让 AI Agent 更懂自己、更能安全执行任务的高信息密度个人用户。

典型场景包括：

- 管理个人笔记、项目资料、合同、证件和家庭文档。
- 追踪日历、任务、承诺和截止日期。
- 维护稳定的人际关系上下文。
- 记录偏好、长期目标、决策原因和历史承诺。
- 为多个 Agent 提供受控的上下文查询能力。

## 产品目标

- 让个人数据可被找到、引用来源、解释来历。
- 让 Agent 在最小权限下访问任务相关上下文。
- 让长期记忆可查看、修改、删除和过期。
- 让高风险动作必须经过用户确认。
- 让系统能从文档库逐步演进到可行动 Agent。

## 第一阶段目标

第一阶段验证 Personal Context Layer，而不是 RAG、单一知识库或 Obsidian 增强器。

Personal Context Layer 的目标是把不同个人数据源转成任务级上下文：

- 上下文来自哪个 Source Adapter。
- 上下文对应哪个 Source Revision。
- 上下文是否仍然 current。
- 上下文是否在当前权限范围内。
- 上下文是否包含 Knowledge Candidate。
- 哪些候选知识需要 User Confirmation 才能进入 Canonical Store 或 Memory。

User Confirmation 不应阻塞普通问答。只有当候选知识要进入长期层、规范事实、自动化规则或外部动作时，才进入确认流程。

## Source-Neutral 边界

LifeMesh 不应围绕某一个数据源设计。Obsidian Vault 是第一个验证适配器，用来验证可编辑知识源的索引、溯源、引用状态和删除级联，但核心概念必须能迁移到日历、任务、邮件、联系人、文件夹和其他个人数据源。

## 非目标

- 不追求一开始全量导入所有个人数据。
- 不把聊天全文、健康、金融、位置等高敏感数据作为早期默认接入对象。
- 不做只靠向量数据库的简单知识库。
- 不让 Agent 拥有不受限制的全域权限。
- 不把 LifeMesh 做成 Obsidian 增强器或单一知识库产品。
- 不把第一阶段收窄成原文片段检索或普通 RAG。
