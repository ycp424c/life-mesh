# User Stories

状态：draft
最后更新：2026-06-26
职责边界：记录 LifeMesh 的候选用户故事，用于从愿景收敛到可实现的最小范围。

## 第 1 阶段：Personal Context Layer

- 作为用户，我希望接入本地 Obsidian Vault 后，可以询问问题并看到引用到具体 Vault Note 的来源。
- 作为用户，我希望系统能遵守路径排除规则，例如不索引 Trash、归档或临时目录中的内容。
- 作为用户，我希望第一版只读索引 Markdown 内容，不修改 Obsidian Vault。
- 作为用户，我希望修改 Obsidian 笔记后，LifeMesh 不再用旧内容回答新问题。
- 作为用户，我希望查看旧回答时能知道它引用的是当时的笔记版本，还是当前仍有效的来源。
- 作为用户，我希望旧回答在来源变更后保留历史内容，但提示我可以基于当前笔记重新生成。
- 作为用户，我希望 Agent 拿到的是带来源、权限、新鲜度和引用状态的 Context Bundle，而不是一堆裸检索片段。
- 作为用户，我希望 Context Bundle 优先使用已核实的 Canonical Fact，其次才是记忆和新鲜来源，候选知识只作为线索而不伪装成事实。
- 作为用户，我希望来源失效时不要静默丢弃，也不要继续当作有效，而是告诉我哪些事实需要复核。
- 作为用户，我希望 Agent 拿到 Context Bundle 时按角色消费：事实回答只引用已核实事实和原文，偏好只影响语气，候选线索一定标注"未核实"，不会把我的偏好或猜测当成客观结论。
- 作为用户，我希望第 1 阶段不引入要维护的运行时服务；Bundle 作为 JSON 产物交付，任何 agent 配合 skill 就能消费，不绑定特定协议。
- 作为用户，我希望系统能提出候选事实、偏好、关系、任务或决策，但在我确认前不把它们当成长期事实。
- 作为用户，我希望每条候选知识都能解释为什么被提出、来自哪些来源、置信度和风险是什么。
- 作为用户，我希望普通回答不要被候选知识确认流程打断。
- 作为用户，我希望候选知识可以先进入收件箱，之后批量接受、忽略、合并或丢弃。
- 作为用户，我希望只有我确认、我手动创建或低风险策略接受的事实，才会成为可复用的 Canonical Fact。
- 作为用户，我希望每条 Canonical Fact 都能看到来源、接受路径、有效性和撤销状态。
- 作为用户，我希望 Memory 只影响 Agent 的语气和排序，不会被当成事实证据；如果某条记忆要当事实用，系统会让我先确认它升级为 Canonical Fact。
- 作为用户，我希望普通偏好推断可以被静默记住并标注不确定性，但重要偏好、关系或决策类推断在写入记忆前先让我确认。
- 作为用户，我希望删除一份文档时，相关索引和派生事实也不再影响回答。

## 第 2 阶段：时间与任务

- 作为用户，我希望 Agent 能根据日历和任务帮我准备每日计划。
- 作为用户，我希望系统能识别文档或消息里的承诺和截止日期，并生成待确认任务。
- 作为用户，我希望修改或撤销一个承诺后，Agent 不再按旧信息提醒我。

## 第 3 阶段：人际上下文

- 作为用户，我希望 Agent 知道某个联系人与哪些项目相关，但不需要读取全部聊天记录。
- 作为用户，我希望关系备注有来源、更新时间和敏感级别。

## 第 4 阶段：长期记忆

- 作为用户，我希望能查看 Agent 记住了什么。
- 作为用户，我希望能删除、修改或让某条记忆过期。
- 作为用户，我希望系统在写入重要推断记忆前先让我确认。

## 第 6 阶段：可行动 Agent

- 作为用户，我希望 Agent 先生成邮件或任务草稿，而不是直接发送。
- 作为用户，我希望每次外部动作都有明确确认和事后审计记录。
